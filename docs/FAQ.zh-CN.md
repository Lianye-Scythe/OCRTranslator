# 常见问题 FAQ

[繁體中文](FAQ.md)｜简体中文｜[English](FAQ.en.md)

## 1. OCRTranslator 目前支持哪些平台？

目前主要面向 **Windows** 桌面使用场景。

虽然部分 Python 代码具备可移植性，但当前的全局快捷键、系统托盘、打包脚本与使用流程都优先围绕 Windows 验证与调整。

## 2. 我需要自己准备 API Key 吗？

需要。

OCRTranslator 不内建模型服务，也不代管任何 API Key。你需要自行准备对应服务的密钥与 Base URL，再填入设置页中的 API Profile。

## 3. 当前支持哪些服务？

当前以：

- `Gemini Compatible`
- `OpenAI Compatible`

这两类接口为主。只要你的服务端 API 兼容，就有机会接入使用。

## 4. 有内建离线 OCR 吗？

当前没有。

当前的设计重点是把截图、选中文本与手动输入流程接到多模态 / 文本模型，因此识别与最终输出质量会明显依赖你实际接入的模型能力。

## 5. 为什么“选中文本”有时在某些应用里不稳定？

选中文本流程目前采用“模拟复制并尽量还原剪贴板”策略，因此：

- 某些自定义 UI 应用可能不响应标准复制快捷键
- 某些游戏、远程桌面或沙盒环境可能拦截快捷键
- 某些程序可能在选中状态或焦点切换上有特殊行为

如果你主要依赖这个入口，建议先在自己常用的目标应用里验证一次。

## 6. 为什么 Release 只提供 ZIP，不直接提供独立 `.exe`？

当前公开发布策略以 **版本化 ZIP** 为主，因为这样可以：

- 保持便携式分发结构
- 一并附带 `README.md`、`LICENSE` 与 `config.example.json`
- 降低用户漏拿必要文件的概率
- 让 Release 资产维持一致且可预期

## 7. 当前公开 Windows 发布包有签名吗？

**当前尚未签名。**

仓库已经预留 SignPath / Trusted Build 所需结构，后续有导入代码签名的规划；在正式导入前，请优先以 GitHub Releases 中的版本化 ZIP 为准。

## 8. 配置文件会存在哪里？

默认会优先使用便携配置文件：

- 源码模式：项目根目录的 `config.json`
- 打包 exe：exe 同层的 `config.json`

如果便携位置当前没有配置文件，而且该运行目录不可写，程序会自动回退到：

- Windows：`%LOCALAPPDATA%\OCRTranslator\config.json`
- 其他环境 fallback：`~/.ocrtranslator/config.json`

只要便携配置文件存在，就会优先使用便携路径；这样可以同时兼顾便携使用与只读目录下的稳定启动。

## 9. 如果我发现安全问题，应该怎么回报？

请不要直接把敏感细节贴在公开 issue 中。

如果问题涉及：

- API Key 泄露
- 私有 Base URL
- 可重放请求
- 可能被滥用的安全弱点

请优先发送邮件到：`po12017po@gmail.com`

## 10. 项目现在适合直接用于正式工作环境吗？

当前更适合定位成：

- 个人使用
- 小规模内部使用
- 愿意接受快速迭代的 public beta / early access 用户

如果你要在正式工作环境中使用，建议至少自行确认：

- 目标模型稳定性
- API Key 管理方式
- 选中文本流程是否符合你的工作软件
- 未签名发布包是否符合你的环境要求

## 11. 为什么有时请求回复很快，有时又会变慢，甚至出现 429 / 503？

请求回复速度会明显受到你实际接入的 AI / LLM 模型、服务端负载、网络状况、配额与官方限流策略影响，并不是 OCRTranslator 单方面可以完全决定。

以当前默认示例里常见的 Google 官方 `gemini-3.1-flash-lite-preview` 为例，体感上通常大约会在 **5～10 秒** 左右返回结果；但在服务较忙、网络较慢或上游排队时，也可能拉长到 **30～40 秒**。

在极端情况下，服务端也可能返回：

- `429 Too Many Requests`
- `503 Service Unavailable`

这通常表示上游限流、暂时不可用，或服务端压力较高。

如果你对回复速度比较敏感，建议至少留意：

- 不同模型之间的延迟差异
- 服务商 / 区域 / 时段带来的波动
- API Key 配额与限流状态
- 是否需要重试、稍后再试，或改用其他模型

## 12. 为什么某些 Compatible 后端会自动改成非流式，或 `Test API` 的表现和正式请求一样？

当前“流式响应”默认是开启的，而 `Test API` 也会沿用相同设置，目的是在正式保存前尽量复现你实际会遇到的后端行为。

如果你接的是第三方 Compatible 后端，而对方的 stream endpoint、SSE chunk 格式或兼容层实现与当前支持的 schema 不一致，OCRTranslator 可能会：

- 先显示流式重试 / 回退提示
- 自动再试一次非流式请求
- 保留当前已收到的部分结果，并标示当前状态

这代表某些不兼容后端在失败回退时，可能会收到 **第二次请求**。如果你的后端长期不适合流式模式，建议直接在进阶设置里关闭“流式响应”。
