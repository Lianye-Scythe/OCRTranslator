# 常见问题 FAQ

[繁體中文](FAQ.md)｜简体中文｜[English](FAQ.en.md)

## 1. OCRTranslator 目前支持哪些平台？

目前主要面向 **Windows** 桌面使用场景。

虽然部分 Python 代码具备可移植性，但当前的全局快捷键、系统托盘、打包脚本与使用流程都优先围绕 Windows 验证与调整。

## 2. 我需要自己准备 API Key 吗？

需要。

OCRTranslator 不内建模型服务，也不代管任何 API Key。你需要自行准备对应服务的密钥与 Base URL，再填入设置页中的 API Profile。

## 3. 当前支持哪些服务？

当前以：

- `Gemini Compatible`
- `OpenAI Compatible`

这两类接口为主。只要你的服务端 API 兼容，就有机会接入使用。

## 4. 有内建离线 OCR 吗？

当前没有。

当前的设计重点是把截图、选中文本与手动输入流程接到多模态 / 文本模型，因此识别与最终输出质量会明显依赖你实际接入的模型能力。

## 5. 为什么“选中文本”有时在某些应用里不稳定？

选中文本流程目前采用“模拟复制并尽量还原剪贴板”策略，因此：

- 某些自定义 UI 应用可能不响应标准复制快捷键
- 某些游戏、远程桌面或沙盒环境可能拦截快捷键
- 某些程序可能在选中状态或焦点切换上有特殊行为

如果你主要依赖这个入口，建议先在自己常用的目标应用里验证一次。

## 6. 为什么 Release 只提供 ZIP，不直接提供独立 `.exe`？

当前公开发布策略以 **版本化 ZIP** 为主，因为这样可以：

- 保持便携式分发结构
- 一并附带 `README.md`、`LICENSE` 与 `config.example.json`
- 降低用户漏拿必要文件的概率
- 让 Release 资产维持一致且可预期

## 7. 当前公开 Windows 发布包有签名吗？

**当前尚未签名。**

仓库已经预留 SignPath / Trusted Build 所需结构，后续有导入代码签名的规划；在正式导入前，请优先以 GitHub Releases 中的版本化 ZIP 为准。

## 8. 配置文件会存在哪里？

默认规则如下：

- 源码模式：项目根目录的 `config.json`
- 打包 exe：exe 同层的 `config.json`

这样做是为了保持便携式使用与搬移便利性。

## 9. 如果我发现安全问题，应该怎么回报？

请不要直接把敏感细节贴在公开 issue 中。

如果问题涉及：

- API Key 泄露
- 私有 Base URL
- 可重放请求
- 可能被滥用的安全弱点

请优先发送邮件到：`po12017po@gmail.com`

## 10. 项目现在适合直接用于正式工作环境吗？

当前更适合定位成：

- 个人使用
- 小规模内部使用
- 愿意接受快速迭代的 public beta / early access 用户

如果你要在正式工作环境中使用，建议至少自行确认：

- 目标模型稳定性
- API Key 管理方式
- 选中文本流程是否符合你的工作软件
- 未签名发布包是否符合你的环境要求
