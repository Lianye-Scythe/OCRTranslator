# 常見問題 FAQ

繁體中文｜[简体中文](FAQ.zh-CN.md)｜[English](FAQ.en.md)

## 1. OCRTranslator 目前支援哪些平台？

目前主要以 **Windows** 桌面使用情境為主。

雖然部分 Python 程式碼具有可攜性，但目前的全域快捷鍵、系統匣、打包腳本與使用流程都優先圍繞 Windows 驗證與調整。

## 2. 我需要自己準備 API Key 嗎？

需要。

OCRTranslator 不內建模型服務，也不代管任何 API Key。你需要自行準備對應服務的金鑰與 Base URL，再填入設定頁中的 API Profile。

## 3. 目前支援哪些服務？

目前以：

- `Gemini Compatible`
- `OpenAI Compatible`

這兩類介面為主。只要你的服務端 API 相容，就有機會接入使用。

## 4. 有內建離線 OCR 嗎？

目前沒有。

目前的設計重點是把截圖、選取文字與手動輸入流程接到多模態 / 文字模型，因此辨識與最終輸出品質會明顯依賴你實際接入的模型能力。

## 5. 為什麼「選取文字」有時在某些應用中不穩定？

選取文字流程目前採用「模擬複製並盡量還原剪貼簿」策略，因此：

- 某些自訂 UI 應用可能不響應標準複製快捷鍵
- 某些遊戲、遠端桌面或沙盒環境可能攔截快捷鍵
- 某些程式可能在選取狀態或焦點切換上有特殊行為

如果你主要依賴這個入口，建議優先在實際常用應用上先做一次驗證。

## 6. 為什麼 Release 只提供 ZIP，不直接提供獨立 `.exe`？

目前公開發佈策略是以 **版本化 ZIP** 為主，因為這樣可以：

- 保持便攜式分發結構
- 一併附帶 `README.md`、`LICENSE` 與 `config.example.json`
- 降低使用者漏拿必要檔案的機率
- 讓 Release 資產維持一致且可預期

## 7. 目前公開 Windows 發佈包有簽名嗎？

**目前尚未簽名。**

倉庫已經預留 SignPath / Trusted Build 所需結構，後續有導入程式碼簽名的規劃；在正式導入前，請優先以 GitHub Releases 中的版本化 ZIP 為準。

## 8. 設定檔會存在哪裡？

預設規則如下：

- 原始碼模式：專案根目錄的 `config.json`
- 打包 exe：exe 同層的 `config.json`

這樣做是為了維持便攜式使用與搬移便利性。

## 9. 如果我發現安全問題，該怎麼回報？

請不要直接把敏感細節貼在公開 issue。

若問題涉及：

- API Key 外洩
- 私有 Base URL
- 可重放請求
- 可能被濫用的安全弱點

請優先寄信到：`po12017po@gmail.com`

## 10. 專案現在適合直接用在正式工作環境嗎？

目前比較適合定位成：

- 個人使用
- 小規模內部使用
- 願意接受快速迭代的 public beta / early access 使用者

如果你要在正式工作環境導入，建議至少自行確認：

- 目標模型穩定性
- API Key 管理方式
- 選取文字流程是否符合你的工作軟體
- 未簽名發佈包是否符合你的環境要求
